
## 3.1 昼夜表

### 起源 — 高中时期的做法

昼夜表最早不是一个软件，是高中时期画在活页纸上的一张手绘表格。

把一天切成很多个五分钟，每个五分钟都值得认真对待。如果这五分钟确实在稳扎稳打地推进一件小事，就给这个格子打一个勾——这个五分钟被定义为**高效的五分钟**。许多高效的五分钟拼在一起，必定是非常富有成效的一整天。量变产生质变。

这张纸上的昼夜表同时是双向的：

- **回顾**：今天做了什么，每段时间在哪件事上，格子里的勾和空白一眼说明一切
- **展望**：把计划写进未来的格子，今天还剩多少时间，距离截止点还有多久，直观可见
- **推算（绝对化指令）**：如果今天有一个计数目标，还剩这么多格子，反推出每分钟至少要完成多少。用彩色笔把剩余空白格框起来，对自己下达铁血 deadline：接下来 XX 分钟必须完成 XX 页，再 XX 分钟完成 XX 页。把终点折算成此刻的节奏，与时间赛跑。

高中时期还有一套**复习标记链**——分为主项目链、次项目链、小项目链三个层级。小项目链用数字串联成一条链状结构，标记一件事的推进过程。这是现在昼夜表里活动分为**总目标 → 活动 → 事件**三层的直接原型，也是当前走线上数字节点链式排列的来源。

### 考研时期的演化

学习转移到电脑上之后，手绘太费力。这一时期做了一个完整的 PyQt 项目——**渐近激励播放器**，用于追踪一天的学习/工作时长，是高中昼夜表系统的数字化集大成。只可惜技术栈选了 PyQt，留有遗憾。

其核心是**渐近激励机制**：累计专注时长到达某个阶段，自动切换新的音乐氛围，触发奖状弹出、全屏彩带特效……用仪式感强化每一个里程碑时刻。这套机制未来可以作为独立模块接入暗影君王系统，属于靠后的实现需求。

渐近激励播放器内部包含：

- **Excel 昼夜表**：还原当初的手绘表格，学习在电脑上完成，记录也在电脑上完成
- **磬声五分钟提醒**：Python 脚本每隔五分钟自动敲一声磬，弹出窗口询问这五分钟在做什么，选择实心或空心五角星，填入说明，写入 Excel
- **活动标注**（源自高中复习标记链）：最初在 Excel 顶部拖透明矩形框住单元格居中文本描述填写，后觉得太麻烦，改为手机脚本自动化控制「时间块」App 完成填写

高中时期**绝对化指令**的思路在考研时期进化成了另一个新项目——**React + Vite 小目标追踪器**（第一个有完整注册登录的项目）。它记录每个历史活动的平均小目标用时，用双秒表的方式与时间赛跑，冲刺每天的工作节奏。绝对化指令的核心没有变：用历史数据校准预期，用实时倒计时制造紧迫感，把模糊的「今天要完成很多」变成「接下来这十分钟必须完成这一件」。

### 当前实现

#### 昼夜表骨架

骨架是一个时间检阅台——5 分钟一个单元格，每列代表一个小时，24 列覆盖一整天。

站在这张表前面，一天的时间被非常舒适地摊开：大概用在了什么上面，效率怎么样，接下来还有多少时间可以调配，哪件紧张的事情需要压缩在哪个时间段内完成——一眼可见。

骨架的收缩与展开不是简单的两档切换，而是一种**究极的响应式设计**：无论屏幕多大多小，昼夜表都能在当前尺寸下呈现最重点的信息。极限情况下浓缩到巴掌大的屏幕，关键信息依然清晰可读。这意味着昼夜表可以永久挂载在屏幕一角，常驻不关闭。

> 极致的响应式设计是暗影君王系统所有面板组件的标配。

#### 双数据源设计

昼夜表原本就有两种视觉元素——**跨单元格高亮矩形**和**竖向走线**。重新设计后，这两种元素的数据源被彻底替换为 ManicTime 的两类数据，视觉结构本身不变，内容来源变了。

##### 跨单元格高亮矩形 ← ManicTime 标签

原本的高亮矩形由手动创建的活动驱动。现在直接读取 ManicTime 的标签数据：用户在 ManicTime 里给一段时间打上标签，这段时间在昼夜表上就以对应颜色的矩形高亮显示。

这一层代表的是**主人对这段时间的主观定义**——我认为这段时间在做什么。标签路径（如"编程, 氛围编程, solo-leveling-system项目"）即为这块高亮的内容描述。

##### 竖向走线 ← ManicTime 应用追踪

原本的走线是细线，标记手动活动的时间范围。现在读取 ManicTime 的自动应用追踪数据，走线**变粗**，颜色完全采用 ManicTime 为每个应用分配的颜色，精确反映这段时间实际运行的程序。

这一层代表的是**客观事实**——电脑上实际发生了什么。

两者叠加，一眼可见主观标注与客观行为的吻合程度。

---

#### 废弃的旧设计

早期昼夜表内置了一套手动活动系统：在时间轴上拖拽创建时间块，填写活动标题、分类、目标对齐，在活动内部标记事件节点。这套系统存在根本性的使用摩擦——专注时手动记录是打扰，事后补录又不准确。

随着 ManicTime 感知底座的引入，这套手动系统被彻底废弃：

- 手动拖拽创建活动的交互 → 删除
- 活动/事件的层级概念 → 删除
- 走线上的步骤节点 → 删除
- 对应的数据库表（`chronos_activities`、`chronos_events`）→ 废弃

记录的责任完全交给 ManicTime，昼夜表只负责展示与解读。

### ManicTime — 感知底座

#### 为什么选择 ManicTime

昼夜表要回答的第一个问题是：**今天做了什么？**

这件事如果靠手动记录，是一种负担。人在专注状态下不会停下来打卡，事后回忆又会丢失细节。真正的感知层需要是被动的、自动的、无打扰的。

ManicTime 是一款本地时间追踪软件，在后台自动记录每一个活跃窗口、每一个应用切换，精确到秒。它解决的正是昼夜表感知层的核心问题——在你完全不操心的情况下，把你今天做了什么如实记录下来。

#### 两层分工

ManicTime 和暗影君王系统各自只做自己最擅长的事：

| 层级 | 负责方 | 内容 |
|------|--------|------|
| 感知层 | ManicTime | 自动追踪应用/窗口、离开检测、截图记录 |
| 意义层 | 暗影君王系统 | 目标对齐、动机锚点、游戏化、AI 洞察 |

ManicTime 是传感器，不是竞品。它的存在让暗影君王系统可以跳过最难的数据采集问题，直接专注在"这些时间意味着什么"上面。

#### ManicTime 标签即活动分类

ManicTime 内置一套多层标签系统，用逗号分隔的路径表示层级关系：

```
编程, 氛围编程, solo-leveling-system项目, 本地去调用ManicTime数据库
时间管理, 软件使用, 软件ManicTime学习使用
生活日常, 家务, 洗碗
```

这个结构与昼夜表的活动分类体系天然吻合。用户在 ManicTime 里打的标签，就是昼夜表的活动分类输入。不需要重复记录，不需要数据同步——直接读取 ManicTime 的本地 SQLite 数据库，始终拿到最新状态。

层级的深度是弹性的。一般 2-3 层足够，也可以更深。最底层的标签是最具体的"正在做的事"，中间层是项目或模式，顶层是大类。

#### 直接读取，不同步

系统通过只读连接按需访问 ManicTime 的本地数据库（`ManicTimeReports.db`），不维护任何副本，不做增量同步。

这样设计的原因：
- ManicTime 的数据随时在变（新增、修改、删除），同步副本必然产生不一致
- WAL 模式下 SQLite 天然支持并发只读访问，不影响 ManicTime 自身运行
- 按需读取永远是最新状态，逻辑最简单

查询范围使用正确的时间窗口：`start_at < 明天 AND end_at > 今天`，确保跨天的长时段记录不被遗漏。
