## 一句话总纲

暗影君王系统不是单纯的时间记录、任务管理或游戏化待办，而是：

> 以个人语境为燃料，以 Fairy 为第二系统，以昼夜表为现实感知面板，把真实行动翻译成“我是否正在靠近真正想要的东西”。

现在应该融合的三条主线是：

1. 动机仪表盘：显示用户真正想要什么、离它多远、最近是在靠近还是远离。
2. 昼夜表任务调度：把任务放入真实剩余时间里，显示还剩多少格、还能怎么安排。
3. 人升式经济激励：用金币、商品、仓库、成就、开箱、合成、限购承接行动后的反馈。

三者不是并列入口，而是同一个系统的三层：

```text
动机仪表盘：决定方向
昼夜表调度：安排今天
经济激励层：强化行动
Fairy：解释、追问、重排、干预
```

## 资料代际

### 第一代：Claude Code 文档原型

原始位置：`D:\my_pro\solo_leveling_system`  
当前归档：[[早期ClaudeCode原型归档]]

这一代是项目最原始的灵魂。它没有完整 App，而是用 Claude Code 对话 + Markdown 文件跑一个“系统”：

- [[产品构想]]：提出“不需要开发 App，直接用对话 + 文档维护，就是一个可运行的系统”。
- [[早期ClaudeCode原型CLAUDE|旧原型 CLAUDE]]：规定 AI 每轮必须先看时间、读任务文件、判断用户现在该干什么。
- [[系统设计v3-动机可视化]]：提出“不是管理时间，是看清欲望”，主页面是动机仪表盘。
- [[系统架构设计]]：提出三层模型：动机 -> 目标 -> 任务。
- [[12月2日工作日志]]：手工版感知层，用聊天时间轴记录编程、等待、分心、生活、消费。

可回收核心：

- AI 不是提醒机器人，而是“懂动机的对话伙伴”。
- 任务必须能回溯到动机，否则只是待办噪音。
- 日志只记录已经发生的事，计划记录将要发生的事。
- 当用户说“待会儿”，AI 要追问“具体几点”。
- 主页面不是任务列表，而是“你在追什么”的动机镜子。

需要注意：

- 早期文档已经主动反思过“经验值/等级/金币可能是假反馈”。所以经济系统可以做，但不应该抢主位。

## 核心模块的最终分工

### 1. 动机仪表盘

回答：

- 我现在真正想要什么？
- 离它还有多远？
- 最近是在靠近还是远离？

示例：

```text
让她见到我时心动      43%   本周 +2%
不再为钱焦虑          18%   本周 -4%
做出自己的产品        51%   本周 +1%
修复重要关系          30%   风险升高
```

数据对象：

- Motivation：诚实欲望。
- Goal：可衡量目标。
- Dimension：目标维度，如体脂、收入、产品进度、关系状态。
- ProgressHistory：趋势历史。

### 2. 昼夜表任务调度

回答：

- 今天已经怎么过了？
- 还剩多少真实可用时间？
- 哪些任务必须塞进剩余格子？
- 如果继续拖延，哪个目标会受损？

任务应具备：

- 关联动机。
- 关联目标。
- 预估耗时。
- 截止时间。
- 可拆分程度。
- 是否需要连续时间。
- 完成后奖励。

示例：

```text
今日剩余可用时间：5h40m
硬截止任务总量：4h10m
缓冲：1h30m
如果 16:30 前不开工，晚间训练会被挤掉。
```

### 3. 人升式经济激励层

回答：

- 完成行动后获得了什么？
- 这些奖励能兑换什么现实反馈？
- 哪些长期成就在被推进？

模块：

- 钱包：金币、特殊货币。
- 商店：现实奖励、AI 权限奖励、休息券、娱乐券。
- 仓库：未使用奖励、收集品、合成材料。
- 成就：条件检测、长期里程碑。
- 开箱/合成/限购：制造期待、稀缺和长期养成。

原则：

- 手动完成任务给基础奖励。
- 感知层验证后给额外奖励。
- 目标对齐度越高，奖励越高。
- 偏离动机的消费或娱乐可以进入惩罚/限购规则。

### 4. Fairy

Fairy 不是一个页面，而是整个系统的运行时解释器。

职责：

- 查询动机、目标、任务、昼夜表、B站、在场状态。
- 主动询问无法判断的窗口意图。
- 将大任务切成“路灯法则”的下一步。
- 当行为偏离动机时进行提醒或重排。
- 从对话和经历里提炼战略 Buff。
- 给奖励系统提供验收与解释。

### 5. Buff / 语境锚点 / 训练系统

回答：

- 我过去获得过哪些真正改变判断的想法？
- 当前情况应该调用哪个方法论？
- 哪些体验值得被提问、复习、内化？

来源：

- 视频、文章、对话、失败、焦虑、项目经验。

形态：

- 战略 Buff：长期认知框架。
- 临时 Buff/Debuff：当前状态影响。
- 训练卡：AI 从经历中生成的问题，进入间隔重复。

## 什么应该保留，什么应该丢掉

### 应该保留

- 动机 -> 目标 -> 任务的三层模型。
- 动机仪表盘作为真正首页。
- 昼夜表的回顾、展望、推算。
- 主动询问与窗口记忆。
- 战略 Buff 库。
- 训练系统：体验 -> 提问 -> 内化。
- 人升式经济系统，但作为激励层。
- Fairy 的“善意管制者”定位。
- 系统在场感。

### 应该弱化

- 虚拟等级作为主反馈。
- 纯暗影君主风格 UI。
- 没有现实意义的属性数值。
- 单纯任务完成率。
- 单纯时间统计。

### 应该丢掉或归档

- 旧 Python server 作为当前架构依据。
- 旧 Swift/macOS 架构里的实现路径。
- 只为模仿《我独自升级》而存在的中二面板。
- “AI 猜一切”的路线。猜不到就问主人。

## 下一步建议

### 第一步：先做“概念账本”

新增或整理以下数据模型，不急着做复杂 UI：

```text
motivations
goals
tasks
schedule_blocks
reward_ledger
shop_items
inventory_items
achievements
strategy_buffs
```

先让 Fairy 能读写这些对象。

### 第二步：复活动机仪表盘

用 mock 数据先做一个主页面原型：

- 3-5 个动机卡片。
- 每个动机显示进度、趋势、截止时间、今日下一步。
- 点击进入维度详情。

不要先做商店，不要先做复杂成就。主页面先定魂。

### 第三步：任务进入昼夜表

让任务带着耗时和 deadline 进入昼夜表侧栏：

- 显示今日剩余可用时间。
- 显示任务总耗时。
- 显示缓冲时间。
- 支持拖入未来时间格。
- Fairy 可以自动建议排程。

### 第四步：接入奖励经济

实现最小闭环：

```text
任务完成 -> 记录奖励 -> 钱包增加 -> 可购买商品 -> 仓库存放/使用 -> 成就进度变化
```

先做可解释、可手动校正，不急着做复杂开箱。

### 第五步：让感知层参与验收

把昼夜表、应用记录、B站记录、在场状态作为“任务完成证据”：

- 手动完成：基础奖励。
- 系统验证：额外奖励。
- 证据不足：Fairy 询问确认。

## 当前最清晰的产品形态

```text
首页：动机仪表盘
  看清真正想要什么，以及最近是否靠近。

今日：昼夜表 + 任务调度
  看清今天已经怎么过，还剩多少时间，任务怎么塞进现实。

奖励：商店 / 仓库 / 成就 / 开箱
  把真实行动变成可感知的反馈和长期养成。

记忆：Buff / Debuff / 语境锚点 / 训练卡
  把经历里的关键表征沉淀为可调用的认知资产。

Fairy：常驻对话与管制层
  负责解释、追问、重排、干预和沉淀。
```

最终要避免的误区：

> 不要让系统变成“暗影君主皮肤的人升”，也不要变成“带聊天的时间追踪器”。  
> 它应该是一个理解个人语境、看守真实动机、调度当日行动、并给现实行动反馈的 AI 第二系统。
