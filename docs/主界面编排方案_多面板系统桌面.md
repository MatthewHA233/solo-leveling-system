# 主界面编排方案：多面板系统桌面

> 这份文档修正一个更底层的交互判断：昼夜表不应该被当成唯一主界面。它很重要，但它只是系统桌面里的一个现实感知面板。

## 关键判断

当前页面空间已经很紧，不能再在昼夜表上方堆一个常驻大主控条。

更准确的主界面模型应该是：

```text
主界面 = 多面板系统桌面
昼夜表 = 现实时间/感知面板
动机仪表盘 = 方向/欲望面板
任务调度 = 今日指令/装载面板
Fairy = 解释/追问/重排面板
经济系统 = 奖励/仓库/成就面板
Buff = 语境记忆/策略面板
```

这意味着产品不应该问：

> 怎么在昼夜表旁边塞更多功能？

而应该问：

> 哪些面板在当前情境下应该占主位，哪些折叠成轨道，哪些交给 Fairy 解释？

---

## 科幻交互隐喻

更适合的隐喻不是“一个大屏幕 + 一个侧栏”，而是舰桥里的多屏工作台：

- 中央屏不永远固定，可以根据模式切换主面板。
- 周边屏显示状态、任务、资源和通讯。
- AI 助手不是一个聊天窗口，而是会在不同面板之间调度注意力。
- 面板可以被“升格”为主屏，也可以缩成摘要卡。

在科幻作品里，真正的系统不是把所有信息塞进一个图，而是提供一套可切换的态势桌面：

```text
态势总览
现实时间
任务指令
目标动机
资源奖励
证据回放
AI 通讯
```

---

## 推荐布局：System Desktop

第一版主界面可以分为三个区域，而不是“昼夜表 + 固定右栏”：

```text
┌────────────────────────────────────────────────────────────┐
│ 顶栏：日期 / 在线状态 / 模型 / 设置                         │
├───────────────┬──────────────────────────────┬─────────────┤
│ 左轨道         │ 主舞台                         │ 右轨道        │
│ 动机摘要       │ 当前主面板                       │ Fairy/详情    │
│ 今日指令       │ 昼夜表 / 动机 / 任务 / 奖励       │ 证据/聊天     │
│ 钱包/Buff      │                              │             │
└───────────────┴──────────────────────────────┴─────────────┘
```

### 左轨道：System Spine

左轨道是系统脊柱，不是完整面板。

它显示：

- 当前主动机。
- 今日最重要任务。
- 剩余可用时间。
- 钱包/奖励状态。
- 激活 Buff。

它的作用是提供常驻方向感，但不要占用太多空间。

### 主舞台：Main Stage

主舞台是可切换的核心区域。

可选主面板：

- `DayNightChart`：现实时间与感知。
- `MotivationDashboard`：动机仪表盘。
- `TaskScheduleBoard`：任务装载和排程。
- `RewardEconomyBoard`：商店、仓库、成就。
- `RetrospectiveBoard`：回顾和证据解释。

昼夜表默认可以是主舞台，但不是永远唯一。

### 右轨道：Fairy / Inspector

右轨道不是固定聊天框，而是上下文解释器。

优先级：

```text
证据详情 > Fairy 对话 > 当前主面板解释 > 活动标签库
```

当用户点击一个 5 分钟格，右轨道解释那段现实。

当用户打开动机面板，右轨道解释动机趋势。

当用户打开任务面板，右轨道解释为什么这样排。

---

## 面板切换不是导航，而是态势模式

不要做传统 tab：

```text
首页 / 今日 / 商店 / 成就 / 设置
```

应该做态势模式：

```text
Reality：看今天真实发生了什么
Directive：看今天还应该做什么
Motivation：看我到底在靠近什么
Reward：看行动反馈和资源
Memory：看 Buff、语境锚点和训练卡
```

这些模式不是网页栏目，而是系统视角。

---

## 昼夜表的新定位

昼夜表仍然是最重要的面板之一，但它的职责要收窄：

昼夜表负责：

- 呈现一天 5 分钟格。
- 显示已发生的活动、应用、B站和在场状态。
- 显示计划任务 overlay。
- 支持点击/hover 某段时间查看证据。
- 支持把任务拖入真实时间。

昼夜表不负责：

- 展示完整动机体系。
- 展示完整商店/仓库/成就。
- 承载所有 Fairy 对话。
- 成为所有信息的唯一容器。

一句话：

> 昼夜表是现实地形图，不是整个作战系统。

---

## 第一版实现策略

不要马上重构整个 App。

建议先做“可逆的布局演进”：

1. 保留当前 `DayNightChart + Right Panel` 结构。
2. 不再增加新的顶部常驻大条。
3. 新增 `SystemDesktopLayout` 作为实验布局组件，但先不替换当前 App。
4. 把 `SystemPilotPanel` 拆成三个未来面板：`MotivationDashboard`、`TaskScheduleBoard`、`RewardEconomyBoard`。
5. 等面板内容稳定后，再决定主界面是否从当前结构迁移到三栏系统桌面。

这样不会破坏当前已完成度很高的 desktop 主线，也不会把核心功能继续塞到右上角。

---

## Stage 0 当前实现修正

上一版尝试把 `System Spine / System Stage` 直接接进当前 App，但实际体验偏抽象，而且会让用户误以为昼夜表旁边又多了一套导航。

当前实现已经改回更具体的方向：

- 暂时不挂载多面板系统桌面。
- 当前 App 仍保持 `DayNightChart + Right Panel` 的稳定结构。
- 在 `DayNightChart` 内部新增悬浮的“昼夜作战台”。
- 原本弱化的小编辑按钮被升级为悬浮面板里的“进入规划 / 填格中”主操作。
- 悬浮面板承担“过去回放、未来规划、剩余空白、当前画笔”的摘要。

当前交互：

```text
昼夜表 = 现实地形图
悬浮昼夜作战台 = 记录/规划控制台
右侧栏 = Fairy / 证据详情 / 标签库
```

这不是最终多面板桌面的完整形态，但它更贴近当前需求：先把昼夜表的过去记录和未来规划操作做顺手。

---

## 对当前 Stage 0 的修正

刚才的 `SYSTEM CORE DECK` 和 `System Spine` 方向都不完全对。

它可保留为视觉素材，但不应该在当前 App 中常驻挂载，因为：

- 会压缩已经紧张的垂直空间。
- 会继续暗示昼夜表是唯一主舞台。
- 会把核心系统变成“昼夜表上方的一条状态栏”，仍然不够自由。

更合适的演进是：

```text
DayNightChart -> 先强化为过去记录 + 未来规划的核心操作面板
TemporalCommandDeck -> 作为昼夜表上方的科幻悬浮控制台
ChatPanel -> 暂时继续作为右侧 Fairy 通讯态
多面板系统桌面 -> 等昼夜表操作体验稳定后再拆分
```

---

## 设计原则

- 主界面是一张系统桌面，不是一张昼夜表。
- 昼夜表是现实地形图，不是所有信息的容器。
- 动机、任务、奖励、Fairy 都应该有自己的面板形态。
- 面板之间要共享同一套数据模型，而不是各做各的。
- 常驻状态应该轻，不要挤压主舞台。
- 核心交互要有科幻感，但不能为了仪式感牺牲可用空间。

最终目标：

> 让用户感觉自己进入了一个可以切换态势视角的 AI 作战桌面，而不是在一个时间图表旁边不断塞小工具。
