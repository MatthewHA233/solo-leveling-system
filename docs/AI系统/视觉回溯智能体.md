> **状态：核心链路已实现**

## 数据链路全览

```
WindowMonitorService（5 秒轮询 + 系统通知）
    ↓ idleSeconds / activityState / windowTitle
CaptureStrategy → AgentManager.captureLoop()
    ↓ 截图（active 期间每 10 秒）
ImageProcessor（压缩 + 缩略图）→ 磁盘 + ScreenshotRecord (SwiftData)
    ↓ 积累截图
批次触发（见下）
    ↓
VideoProcessingService → MP4 视频 + frameTimestamps[]
    ↓
ContextAdvisor.buildContextHint() → 上下文字符串
    ↓
Phase 1：AIClient.streamTranscribeVideo()（qwen-vl-max）
    ↓ [[startSecond, endSecond, description]]
mapSecondsToTimestamps()
    ↓ [[startTs, endTs, startTime, endTime, description]]
Phase 2：AIClient.streamGenerateActivityCards()（qwen-plus）
    ↓ [[category, title, summary, detailedSummary, ...]]
ActivityCardRecord (SwiftData) → 昼夜表展示 + ExpEngine
```

## 截图捕捉

**触发逻辑**（`CaptureStrategy.inferState()`）：

| 条件 | 状态 | 截图间隔 |
|------|------|---------|
| `isScreenLocked` | locked | 停止 |
| `idleSeconds >= 300` | deepIdle | 停止 |
| `idleSeconds >= 60` | idle | 停止 |
| 刚切换窗口（`windowJustSwitched`） | active | 1 秒后立即截 |
| 其余 | active | 10 秒 |

> idle/deepIdle 直接停止而非降频——停止期间画面也是重复的，截了是浪费。

**每次截图做的事**（`AgentManager.performCapture()`）：
1. 隐私检查（屏蔽敏感应用/密码输入框）
2. `RuleClassifier.classify()` 即时分类，emit `contextAnalyzed` 事件驱动游戏引擎
3. 截图 → `ImageProcessor.compress()`（最宽 1280px，JPEG 0.6）+ 缩略图（320px，JPEG 0.5）
4. 写入磁盘：`~/.config/solo-agent/screenshots/{YYYY-MM-DD}/HH-mm-ss_AppName_id.jpg`
5. 写入 `ScreenshotRecord`（SwiftData）

**存储清理**：截图文件 48 小时后自动删除，元数据 7 天后清理。

## 批次触发

三条路径都收敛到 `BatchManager.processCurrentSession()`：

**路径 A：活跃期结束（主要路径）**
`captureLoop()` 检测到 `interval <= 0`（从 active → idle/locked），立即调用 `triggerSessionBatch()`。用户停下来那一刻就开始分析。

**路径 B：定时切割（防止单批过长）**
从开始录屏计时，到达 `batchTargetDuration = 300 秒`（5 分钟）自动切割，重置计时器继续。

**路径 C：定期安全网**
后台每 60 秒轮询一次，有积压未处理的截图就触发。

**最小批次要求**：截图数 `>= max(5, frameStride × 3)`，时长 `>= 150 秒`（2.5 分钟）。

## 视频合成（VideoProcessingService）

截图 → MP4，关键参数：
- 帧率：`config.videoFps`（默认 1fps，即 1 秒一帧）
- 帧采样：`frameStride`（每 N 张截图取 1 帧，减少视频大小）
- 分辨率：`maxHeight`（默认 720p）
- 输出：`~/.config/solo-agent/timelapses/{YYYY-MM-DD}/timelapse_HH-mm-ss.mp4`
- 同时返回 `frameTimestamps: [Int]`（每帧对应的真实 Unix 时间戳，供后续映射）

## 上下文感知（ContextAdvisor）

`buildContextHint(startTs:endTs:config:)` 在 Phase 1 发请求前调用，输出一段字符串注入 prompt。

**信号源（按输出顺序）**：

| 信号 | 来源 | 说明 |
|------|------|------|
| 活跃应用 | `ActivityRecord.appName` | 出现频次 Top 5 |
| 窗口关键词 | `ActivityRecord.windowTitle` | 高频词 Top 8（停用词过滤）|
| 活动类别分布 | `RuleClassifier.classify()` | Top 3 category（无 AI 成本）|
| 用户修正规则 | `ContextRuleStore`（持久化） | 用户手动定义的 pattern → interpretation |
| 操作状态时间线 | `idleSeconds` + `activityState` 推断 | 见下方阈值表 |
| 应用切换时间线 | 检测相邻记录的 appName 变化 | `[HH:mm:ss] → App: 窗口标题` |
| 主线目标 | `config.mainQuest` | 直接注入 |

**操作状态推断阈值**：

| 条件 | 状态 | 含义 |
|------|------|------|
| `activityState == "active"` 且 `idle < 3s` | 主动操作 | 正在打字/点击 |
| `activityState == "active"` 且 `idle < 15s` | 阅览 | 在看内容，偶尔滚动 |
| `activityState == "active"` 且 `idle < 60s` | 等待 | 等 AI 返回/编译/加载 |
| 其余 | 放置 | 人可能离开 |

> 阈值依据：`<3s` 是键盘事件间隔上限；`15s` 是读完一屏文字的时间；`60s` 是等待类操作上限。

查看当前上下文：`/context` 命令。管理用户规则：`/rule add/list/remove`。

## Phase 1：视频转录

**调用**：`AIClient.streamTranscribeVideo()`（流式 SSE，Gemini fallback 非流式）

**Prompt 构建**（`PromptTemplates.videoTranscriptionPrompt()`）：
- 帧→时间映射表：`buildFrameTimeMapping()` 生成「第 N 秒 = HH:mm:ss」，让 AI 在 description 里标注真实时间
- contextHint：ContextAdvisor 提供的全部信号
- 要求 AI 在 description 用【操作】【等待】【阅览】【放置】标注用户状态

**输出**：
```json
[
  { "startSecond": 0, "endSecond": 12, "description": "[10:30] 【操作】在 Cursor 编辑 BatchManager.swift..." },
  { "startSecond": 12, "endSecond": 30, "description": "[10:32] 【等待】Terminal 等待编译结果..." }
]
```

**后处理**：`mapSecondsToTimestamps()` 把 AI 返回的视频秒数转成真实 Unix 时间戳（`sec × fps = 帧索引`）。转录结果缓存到 `batch.transcriptionJson`，支持 `/regenerate` 跳过 Phase 1 重跑。

## Phase 2：活动卡片生成

**调用**：`AIClient.streamGenerateActivityCards()`（流式 SSE）

**Prompt 关键内容**（`PromptTemplates.activityCardPrompt()`）：
- Phase 1 转录 JSON（含真实时间戳和操作状态标记）
- 今日已有卡片（最近 5 张，含 detailedSummary 供衔接判断）
- 主线目标 + 动机 → AI 判断 `goalAlignment`
- contextHint
- **操作密度感知规则**：高密度操作 → "密集开发…"；等待+阅览交替 → "通过 AI 辅助开发…"；大段放置 → 不计入有效工作时间
- **衔接规则**：如果当前转录是已有卡片的延续，返回 `[]`；拓展子任务生成新卡片；完全不同任务才新建

**输出字段**：`category / subcategory / title / summary / detailedSummary / startTs / endTs / goalAlignment / distractions / appSites`

> `startTime`/`endTime` 始终从 `startTs`/`endTs` 按本地时区计算，不用 AI 返回的字符串（AI 不知道用户时区）。

## 本地规则分类（RuleClassifier）

不依赖 AI，两个使用场景：

**1. 实时截图时**：每次 `performCapture()` 立即分类 → emit `contextAnalyzed` → ExpEngine 实时计算 EXP、驱动游戏 buff。

**2. ContextAdvisor 构建时**：聚合批次内所有记录的分类分布，注入 contextHint。

**分类逻辑**（优先级从高到低）：
1. App 名匹配 `windowRules`（confidence 0.8）
2. 若是浏览器，用窗口标题匹配 `browserTitleRules`（confidence 0.75）
3. 窗口标题模糊匹配（confidence 0.6）
4. unknown（confidence 0.3）

分类结果同时给出 `focusScore`（coding=0.8, social=0.3, idle=0.0 等），用于 EXP 加权。
