# 日内相位引擎（草案）

> 状态：草案。  
> 这份文档用于修正“动机仪表盘是否应该作为常驻首页”的判断。当前更接近的方向是：动机不应该全天占据前台，而应该作为后台重力，由系统根据时间、状态和任务压力决定何时显现。

## 1. 为什么不做常驻动机仪表盘

之前的思路是把“动机仪表盘”作为主页面，让用户持续看见：

```text
我真正想要什么
我现在离它多远
最近是否在靠近
```

这个方向有一部分是对的：系统必须理解动机，否则任务、奖励和管制都会变成空壳。

但“动机常驻前台”本身可能是错的。

动机是很重的东西。它如果全天浮现，会让人一直处在自我审视、自我评估、自我追问里，反而分心。人不是每个时间段都适合面对“我到底想成为谁”。早晨可能更需要身体、秩序和韧性；下午才适合发现、兴趣和语境采集；晚上更适合回收、整理和反思；深夜则应该保护睡眠和降低刺激。

因此，动机更像地下水，而不是广告牌。

它应该在后台决定方向，在合适的相位里被系统提取出来，而不是全天变成一个大屏幕盯着用户。

---

## 2. 什么是日内相位

日内相位，是系统对“一天中当前应该处于什么模式”的判断。

它不是传统意义上的导航页面，也不是单纯按时间切换主题色。它更像一套运行协议：

```text
当前是什么时间
用户处于什么状态
基础任务是否完成
是否有 deadline 压力
是否已经疲劳
是否正在偏离主线
是否应该开放探索
是否应该收束和保护
```

系统根据这些信息决定：

```text
此刻前台显示什么
此刻允许用户做什么
此刻 Fairy 应该说什么
此刻哪些设备和内容应该被限制
此刻哪些语境可以被打开
```

第一版可以先把相位理解成“时间给默认值，状态可以覆盖”。

```text
时间给默认相位：
早晨默认韧性
下午默认发现
晚上默认回收
深夜默认保护

状态可以覆盖相位：
基础任务没完成，下午也不能完全开放发现
deadline 压顶，发现相位被压缩
疲劳过高，系统提前进入保护
强语境出现，系统允许临时打开忆泡回看
```

这比“固定首页”更接近暗影君王系统的核心：系统不是展示一堆功能，而是在不同时间接管不同层面的现实。

## 3. 初始相位划分

下面只是第一版语言，不是最终命名。

### 3.1 韧性相位

典型出现时间：早晨，或一天重新启动时。

核心问题：

```text
身体和秩序有没有立起来？
今天的最低生存线有没有完成？
设备是否需要先被管住？
```

前台内容：

```text
每日基础任务
身体状态
晨间训练
洗漱、整理、出门、吃饭等基础秩序
今日硬 deadline 摘要
设备管制状态
```

不应该前台展示：

```text
兴趣发现流
复杂动机分析
短视频式语境回看
商店、开箱、成就娱乐反馈
```

Fairy 的语气：

```text
先把身体和秩序立起来。
探索下午再开放。
现在不是谈理想的时候，是把第一块地基打稳。
```

系统动作：

```text
强制管制娱乐设备
限制短视频、游戏、社交媒体
只显示低歧义的每日任务
必要时启用倒计时和硬提醒
```

### 3.2 执行相位

典型出现时间：上午到下午的主要工作块，或 deadline 压力升高时。

核心问题：

```text
今天最重要的任务是否已经进入现实时间？
剩余 5 分钟格够不够？
是否需要删减、降级或重排？
```

前台内容：

```text
昼夜表
计划安排
实际记录
待排任务
剩余可用时间
任务预计耗时
deadline 倒计时
```

不应该前台展示：

```text
过多语境回忆
开放式发现入口
复杂奖励系统
无关动机自省
```

Fairy 的语气：

```text
这件事要塞进哪几个 5 分钟格？
如果现在不开工，晚上会挤掉什么？
你说“待会儿”，具体几点？
```

系统动作：

```text
把任务换算成 5 分钟格
检查今日剩余容量
推荐可用时间段
在昼夜表上显示计划块
根据实际记录和感知层回填执行情况
```

### 3.3 发现相位

典型出现时间：下午、低压力窗口、基础任务完成之后。

核心问题：

```text
今天允许探索什么？
哪些内容可能成为新的语境？
哪些兴趣值得被收集，而不是被立刻压制？
```

前台内容：

```text
语境小球
忆泡长廊
短视频式回看
B站、文章、对话、截图、窗口片段
兴趣类任务
可收集的 Buff 雏形
```

这里的“短视频感”不是为了制造信息流，而是因为语境回忆本身往往是多模态的。一个想法可能来自一段视频、一句话、一个画面、一段聊天、一种情绪，单纯文字列表无法还原它被点亮时的感觉。

但发现相位必须被边界约束。

它不是无限刷内容，而是有系统许可的探索窗口：

```text
有开始时间
有结束时间
有收集目的
有回收机制
```

Fairy 的语气：

```text
这段内容为什么吸引你？
它和哪个动机有关？
这是新 Buff，还是只是刺激？
要不要把它收成一个忆泡？
```

系统动作：

```text
允许内容探索
收集多模态语境片段
为片段打上临时表征
标记可能关联的动机、Buff、任务
到点后切入回收相位
```

### 3.4 回收相位

典型出现时间：晚上，或探索/执行结束之后。

核心问题：

```text
今天经历了什么？
哪些语境值得留下？
哪些行动靠近了动机？
明天应该怎么安排？
```

前台内容：

```text
昼夜表复盘
实际记录 vs 计划安排
语境小球整理
动机/Buff 提炼
今日奖励结算
明日预排
Fairy 复盘问题
```

回收相位是“动机显现”的主要时刻。

动机不应该在早晨一直压迫用户，但晚上回收时，系统可以把当天的行动、偏离、探索和语境重新汇聚成一句判断：

```text
今天你更靠近了什么？
今天你逃避了什么？
今天哪个语境反复发光？
明天最小的一步是什么？
```

Fairy 的语气：

```text
今天真正有价值的不是完成了多少格，而是哪几个格改变了方向。
这段视频和你“做出自己的产品”的动机有关，我建议收成一个 Buff。
明天早晨不要继续想这个，先做韧性任务。
```

系统动作：

```text
生成当天语境摘要
沉淀 Buff / Debuff / context anchor
更新动机趋势
结算奖励流水
预生成明日基础任务和候选任务
```

### 3.5 保护相位

典型出现时间：深夜、疲劳过高、连续偏离、情绪风险升高时。

核心问题：

```text
系统是否应该降低刺激？
用户是否需要被保护，而不是被继续推动？
哪些设备和内容应该关闭？
```

前台内容：

```text
睡眠倒计时
低刺激提示
必要事项
明日安心卡
设备锁定状态
```

不应该前台展示：

```text
动机审判
复杂复盘
短视频回看
商店奖励
高刺激发现内容
```

Fairy 的语气：

```text
现在不继续追问意义。
保存现场，明天再处理。
你的任务是睡觉，不是把人生想明白。
```

系统动作：

```text
限制高刺激应用
关闭发现入口
只保留睡眠/必要任务
把未处理语境暂存到明日回收
```

## 4. 语境、动机、任务、管制之间怎么流动

日内相位引擎不是替代动机系统，而是决定动机何时出场。

可以暂时理解成下面这条流：

```text
现实经历
  ↓
感知层捕获：窗口、应用、B站、截图、对话、在场状态
  ↓
发现相位：语境小球被收集
  ↓
回收相位：语境小球被整理、归档、提炼
  ↓
动机 / Buff / Debuff / context anchor 被更新
  ↓
执行相位：动机转化成任务和计划块
  ↓
昼夜表：任务进入 5 分钟格
  ↓
感知层和实际记录验收
  ↓
奖励流水 / 管制规则 / Fairy 记忆更新
  ↓
第二天的相位策略被调整
```

这里有几个关键判断：

- 动机不是手填目标列表，而是从语境、经历和反复出现的表征里凝结出来。
- 语境不是全天开放的信息流，而是在发现相位被允许、在回收相位被整理。
- 任务不是独立待办，而是动机在某个相位里的行动化结果。
- 管制不是惩罚用户，而是保护某些相位不被错误内容污染。
- Fairy 不是聊天入口，而是相位之间的解释器、守门员和调度者。

## 5. 对当前界面的影响

这份草案会改变“动机仪表盘作为首页”的设计判断。

当前更合理的方向可能不是：

```text
首页 = 动机仪表盘
```

而是：

```text
首页 = 当前相位面板
```

当前相位面板根据系统判断切换内容：

```text
韧性相位 -> 每日任务 / 管制 / 身体 / 秩序
执行相位 -> 昼夜表 / 任务排程 / 剩余时间
发现相位 -> 语境小球 / 忆泡长廊 / 兴趣探索
回收相位 -> 复盘 / 动机提炼 / Buff 沉淀 / 明日计划
保护相位 -> 睡眠 / 降刺激 / 设备限制
```

动机仪表盘仍然存在，但它不应该全天常驻。它更可能出现在：

```text
回收相位
周复盘
重大偏离时
任务冲突时
Fairy 需要解释“为什么现在要做这个”的时候
```

换句话说：

> 动机是系统判断的依据，不是全天候展示的内容。

## 6. 第一版可以怎么落地

第一版不需要立刻实现复杂自动判断。

可以先做一个极简的 `Phase` 概念：

```ts
type DayPhase =
  | 'resilience'  // 韧性
  | 'execution'   // 执行
  | 'discovery'   // 发现
  | 'recovery'    // 回收
  | 'protection'  // 保护
```

每个相位先定义：

```text
显示哪些面板
隐藏哪些面板
允许哪些任务
限制哪些设备
Fairy 使用哪套语气和规则
是否允许语境流打开
是否允许奖励娱乐反馈打开
```

第一版可以手动切换相位，也可以按时间给默认值：

```text
05:00 - 10:00 韧性
10:00 - 17:30 执行
17:30 - 20:30 发现
20:30 - 23:30 回收
23:30 - 05:00 保护
```

这些时间只是占位，不代表最终方案。

真正重要的是：相位决定前台，而不是动机决定前台。

## 7. 待讨论问题

- [ ] 相位应该叫“相位”“协议”“模式”“时段”，还是别的词？
- [ ] 早晨的核心是“韧性”，还是“秩序”“基础任务”“身体启动”？
- [ ] 下午的“发现相位”是否必须以完成基础任务为解锁条件？
- [ ] 晚上的“回收相位”是否应该自动打开语境小球整理？
- [ ] 深夜保护相位允许用户手动突破吗？突破是否需要记录原因？
- [ ] Fairy 在不同相位是否应该使用不同系统提示词？
- [ ] 设备管制是相位规则的一部分，还是独立的护栏系统？
- [ ] 动机仪表盘在什么情况下可以主动弹出？
- [ ] 语境小球最终沉淀为 `motivation`、`buff`、`task` 的规则是什么？
- [ ] 昼夜表顶部是否应该显示当前相位，而不是显示动机？

## 8. 当前结论

目前最接近的产品判断：

```text
暗影君王系统不是常驻动机仪表盘。
它是一套根据日内相位调度现实、语境、任务、管制和 Fairy 的系统。

动机在后台提供重力。
语境在合适的时间被收集和回收。
任务在执行相位进入昼夜表。
管制保护某些相位不被污染。
Fairy 负责解释为什么此刻应该这样。
```

如果这条线成立，后续主页面要优先设计的不是“动机卡片”，而是：

```text
当前相位卡
相位规则
相位切换
相位中的任务与内容许可
相位如何调用动机和语境
```

这会让系统从“展示我想要什么”变成“在正确时间把正确的我叫出来”。
