# 下一阶段蓝图：动机、昼夜表与经济闭环

> 这份文档是基于 [[产品考古索引]]、[[主界面编排方案_多面板系统桌面]]、当前 Tauri Desktop 主线、[[2026-04-13-daynightchart-responsive-design]]、早期 Claude Code 原型和人升参考文档整理出的下一阶段自由发挥蓝图。

## 一句话判断

下一阶段不要做“暗影皮肤的人升”，也不要做另一个待办应用。应该做的是：

> 让动机决定方向，让任务进入昼夜表，让真实行动被感知层验收，让奖励经济强化行动，让 Fairy 把这一切解释成“我是否正在靠近真正想要的东西”。

```text
Motivation 决定方向
Goal 定义距离
Task 生成下一步
ScheduleBlock 落进现实时间
Evidence 证明发生过什么
RewardLedger 结算反馈
Shop / Inventory / Achievement 承接长期养成
Buff / FairyContext 让 AI 越来越懂主人
```

## 当前现实

当前主线是 `clients/desktop`，也就是 Tauri 2 + React + Rust 桌面端。

交互修正：

> 昼夜表不是唯一主界面，而是多面板系统桌面中的“现实时间/感知面板”。动机、任务、奖励、Fairy 和 Buff 都应该拥有自己的面板形态，而不是继续被塞进昼夜表旁边的小入口。

主要入口：

- `clients/desktop/src/main.tsx`：按 hash 切换主窗口、Fairy、Camera、Debug 窗口。
- `clients/desktop/src/App.tsx`：当前主界面，暂时仍是昼夜表 + 右侧栏结构，但下一阶段不应继续往这个结构里硬塞核心功能。
- `clients/desktop/src/components/DayNightChart.tsx`：昼夜表核心画布，5 分钟格、活动块、感知 span、B站 span 都在这里可视化。
- `clients/desktop/src/lib/local-api.ts`：前端本地 HTTP API 客户端。
- `clients/desktop/src-tauri/src/api.rs`：Rust Axum API 路由。
- `clients/desktop/src-tauri/src/db.rs`：SQLite 表结构和查询。

已有可复用能力：

- `goals` 表已经存在，但字段很轻：`id/title/status/tags/created_at/completed_at`，适合做兼容层，不适合直接承担完整动机仪表盘。
- `activity_blocks` 已经是 5 分钟稀疏块，非常适合承载“任务排进昼夜表”的现实落点。
- `perception_spans`、`bili_spans`、`presence_spans` 已经能成为任务验收证据。
- `clients/desktop/src/lib/game/*` 已经有 `Quest`、`Player`、`ExpEngine`、`QuestEngine`、`PlayerManager`，但基本未接入 UI 和持久化，只能作为规则参考。

关键结论：

> 任务排程不要另造一条时间轴，应该映射到昼夜表的 5 分钟格。奖励经济不要只存在前端内存里，应该先有不可变账本。

## 产品形态

### 首页：动机仪表盘

首页不要先展示任务列表，而是回答三个问题：

- 我真正想要什么？
- 我现在离它多远？
- 最近这些行动是在靠近，还是在远离？

第一版可以只做 3 到 5 张动机卡片：

```text
动机标题
为什么重要
关联目标数
今日下一步
最近 7 天靠近/远离趋势
Fairy 的一句解释
```

参考：[[系统设计v3-动机可视化]]、[[产品构想]]、[[动机系统]]、[[第1章_产品愿景]]

### 今日：昼夜表 + 任务调度

今日页的核心仍然是昼夜表。任务列表不应该独立悬浮，而应该问：

- 今天还剩多少 5 分钟格？
- 待做任务总耗时是多少？
- 哪些任务能塞进去？
- 哪些任务必须降级、推迟或放弃？

第一版可以做一个“待排任务侧栏”：

```text
任务标题
预计耗时
截止时间
关联动机/目标
是否已经排入昼夜表
Fairy 推荐排入的时间段
```

参考：[[第3章_系统面板设计]]、[[昼夜表/README|昼夜表 README]]、[[数据架构]]、[[布局与交互]]、[[视觉风格]]、[[2026-04-13-daynightchart-responsive-design]]

### 奖励：经济激励层

经济系统只做行动后的反馈，不抢主位。

第一版闭环：

```text
任务完成 -> RewardLedger 写入 -> 钱包增加 -> 商店购买 -> 仓库存放/使用 -> 成就进度变化
```

暂缓：

```text
复杂开箱
复杂合成
复杂技能树
大而全属性面板
暗影君王风格首页
```

参考：[[人升官方文档索引]]、[[03_事项|事项]]、[[04_属性|属性]]、[[05_商品|商品]]、[[02_成就|成就]]

### Fairy：系统解释器

Fairy 不是聊天入口，而是系统解释器和第二系统执行层。

每次响应前至少应该拿到：

```text
当前时间
今日剩余可用时间
当前动机
当前目标
今日任务
最近一小时活动
当前窗口/应用/B站/在场状态
钱包与最近奖励流水
激活 Buff
用户规则
```

Fairy 的第一批硬规则：

- 任务没有动机归属时，追问“这是为了什么？”
- 用户说“待会儿”，追问“具体几点？”
- 任务无法塞进今日剩余格子时，要求删减、降级或重排。
- 证据不足时，先询问，不乱判。
- 手动完成给基础奖励，感知层验证给额外奖励。
- 偏离主线时，引用动机和 Buff，而不是泛泛提醒。

参考：[[早期ClaudeCode原型CLAUDE]]、[[主动询问与窗口记忆]]、[[视觉回溯智能体]]、[[PROMPT-ENGINEERING]]

## 最小数据模型

### Motivation

动机是“为什么”。它不是目标，不是任务，也不是标签。

```ts
interface Motivation {
  id: string
  title: string
  why: string
  priority: number
  status: 'active' | 'paused' | 'archived'
  createdAt: string
  updatedAt: string
}
```

### Goal

目标是“距离”。它应该能度量，哪怕第一版允许手动修正。

```ts
interface Goal {
  id: string
  motivationId: string
  title: string
  metricName: string
  currentValue: number
  targetValue: number
  unit: string
  deadline: string | null
  confidence: number
  status: 'active' | 'completed' | 'abandoned'
  createdAt: string
  updatedAt: string
}
```

当前已有 `goals` 表可以短期兼容，但长期建议迁移或扩展出 `motivation_id/metric/deadline/confidence`。

### Task

任务是“下一步行动”。必须能回到目标和动机。

```ts
interface Task {
  id: string
  motivationId: string
  goalId: string
  title: string
  description: string
  status: 'inbox' | 'planned' | 'doing' | 'done' | 'skipped' | 'expired'
  estimatedMinutes: number
  deadline: string | null
  importance: number
  difficulty: 'E' | 'D' | 'C' | 'B' | 'A' | 'S'
  rewardPolicyId: string | null
  evidencePolicy: 'manual' | 'sensor_enhanced' | 'sensor_required'
  createdAt: string
  updatedAt: string
}
```

### ScheduleBlock

排程块是“现实时间”。任务不等于排程块；任务可以计划，排程块表示已安排或已发生。

```ts
interface ScheduleBlock {
  id: string
  taskId: string | null
  activityBlockRef: string | null
  startAt: string
  endAt: string
  source: 'planned' | 'sensed' | 'manual' | 'fairy'
  category: string
  title: string
  motivationId: string | null
  goalId: string | null
  alignmentScore: number | null
  evidenceRefs: string[]
}
```

第一版可以不建完整 `schedule_blocks` 表，而是用 `task_schedule_blocks` 关联 `tasks` 和现有 `activity_blocks` 的 `date/minute`。

### RewardLedger

奖励流水是不可变账本。钱包余额、成就进度都可以从这里推导或缓存。

```ts
interface RewardLedger {
  id: string
  actionType: 'task_done' | 'task_skipped' | 'item_purchase' | 'item_use' | 'achievement_unlock'
  refType: 'task' | 'shop_item' | 'inventory_item' | 'achievement' | 'manual'
  refId: string
  currencyType: 'coin' | 'token' | 'custom'
  amount: number
  reason: string
  evidenceLevel: 'manual' | 'inferred' | 'verified'
  createdAt: string
}
```

### ShopItem

商品不是只给快乐，它也是行为塑形工具。

```ts
interface ShopItem {
  id: string
  title: string
  description: string
  price: number
  currencyType: 'coin' | 'token' | 'custom'
  itemType: 'real_reward' | 'app_effect' | 'collectible' | 'material' | 'permission'
  effectType: 'none' | 'grant_coin' | 'grant_buff' | 'loot_box' | 'custom'
  stockLimit: number | null
  purchaseLimitRule: 'none' | 'daily' | 'weekly' | 'monthly'
  purchasable: boolean
  status: 'listed' | 'unlisted' | 'archived'
}
```

### InventoryItem

仓库承接“获得但未使用”的奖励。

```ts
interface InventoryItem {
  id: string
  shopItemId: string
  quantity: number
  acquiredFromType: 'purchase' | 'task' | 'achievement' | 'system'
  acquiredFromId: string | null
  status: 'stored' | 'used' | 'refunded' | 'expired'
  expiresAt: string | null
}
```

### Achievement

成就是长期里程碑，不是弹窗奖牌。

```ts
interface Achievement {
  id: string
  title: string
  description: string
  conditionType: 'task_count' | 'goal_progress' | 'streak' | 'ledger_sum' | 'custom_query'
  conditionConfig: Record<string, unknown>
  progress: number
  unlockedAt: string | null
  rewardPolicyId: string | null
  status: 'active' | 'unlocked' | 'hidden' | 'archived'
}
```

### Buff

Buff 是可复用的认知资产，不只是加数值。

```ts
interface Buff {
  id: string
  title: string
  content: string
  type: 'strategy' | 'temporary' | 'debuff' | 'training'
  sourceType: 'conversation' | 'video' | 'article' | 'behavior' | 'manual'
  sourceRef: string | null
  triggerCondition: string
  confidence: number
  activationStatus: 'draft' | 'active' | 'retired'
  usageCount: number
  lastUsedAt: string | null
}
```

参考：[[Buff库]]、[[战略Buff库]]、[[训练系统]]

## 技术落地路线

### 阶段 0：只做数据壳和 mock UI

目标：不碰复杂自动化，先让系统能看见“动机 -> 目标 -> 任务 -> 时间 -> 奖励”。

建议新增：

```text
clients/desktop/src/lib/system-models.ts
clients/desktop/src/components/MotivationDashboard.tsx
clients/desktop/src/components/TaskSchedulePanel.tsx
clients/desktop/src/components/RewardEconomyPanel.tsx
```

阶段 0 可以只用 mock 数据，不写数据库。

完成标准：

- App 里能切到动机仪表盘。
- 能看到 3 个动机、关联目标、今日任务、剩余时间提示。
- 不接商店也没关系，但要预留奖励流水的位置。

### 阶段 1：后端持久化

在 `clients/desktop/src-tauri/src/db.rs` 里新增表：

```sql
motivations
tasks
task_schedule_blocks
reward_ledger
shop_items
inventory_items
achievements
buffs
```

在 `clients/desktop/src-tauri/src/api.rs` 增加：

```text
/api/motivations
/api/tasks
/api/tasks/:id/schedule
/api/rewards/ledger
/api/shop/items
/api/inventory
/api/achievements
/api/buffs
```

在 `clients/desktop/src/lib/local-api.ts` 增加同风格客户端方法。

完成标准：

- 任务可以创建、更新、完成。
- 任务可以绑定到动机和目标。
- 完成任务会写入奖励流水。

### 阶段 2：任务进入昼夜表

不要新建独立时间轴。任务排程应该复用昼夜表。

前端插入点：

- `App.tsx`：维护 `tasks/todayTasks/scheduledTaskBlocks` 状态。
- `TaskSchedulePanel.tsx`：显示待排任务、今日剩余容量、推荐时间段。
- `DayNightChart.tsx`：增加 planned task overlay，区别于已发生 activity blocks。
- `local-api.ts`：新增 `scheduleTaskBlocks`、`fetchTasksForDate`。

后端插入点：

- `db.rs`：`task_schedule_blocks` 保存任务和 `date/minute` 的关系。
- `api.rs`：新增任务排程 API。

第一版交互：

```text
点击任务 -> 选择“排入今天”
Fairy 推荐 2 到 3 个时间段
用户确认 -> 昼夜表出现计划块
实际执行后 -> activity_blocks / perception_spans 作为证据
```

完成标准：

- 今日剩余格子可计算。
- 待排任务总耗时可计算。
- 任务是否装得下今天可判断。
- 计划块和实际块能同时显示。

### 阶段 3：奖励经济最小闭环

先做钱包、商店、仓库，不做复杂系统。

前端新增：

```text
RewardEconomyPanel
WalletSummary
ShopList
InventoryList
AchievementStrip
```

后端新增：

```text
reward_ledger
shop_items
inventory_items
achievements
```

完成标准：

```text
任务完成 -> reward_ledger +coin
钱包余额变化
购买商品 -> reward_ledger -coin
inventory_items 增加
使用商品 -> inventory_items 状态变化
achievement progress 更新
```

### 阶段 4：Fairy 上下文注入

Fairy 每次回答前应该拿到一个明确的上下文快照，而不是临时拼 prompt。

建议新增：

```text
clients/desktop/src/lib/ai/fairy-context.ts
```

最小结构：

```ts
interface FairyContextSnapshot {
  now: string
  selectedDate: string
  remainingScheduleCapacity: number
  activeMotivations: Motivation[]
  activeGoals: Goal[]
  todayTasks: Task[]
  recentScheduleBlocks: ScheduleBlock[]
  currentWindowContext: unknown
  presenceState: unknown
  walletSummary: { coin: number }
  recentRewards: RewardLedger[]
  activeBuffs: Buff[]
  userRules: string[]
}
```

完成标准：

- Fairy 能解释为什么推荐某个任务。
- Fairy 能说“今天装不下，必须降级/推迟/删除”。
- Fairy 能根据证据不足追问。
- Fairy 能引用 Buff，而不是泛泛鸡汤。

## 与 superpowers 规格的关系

[[2026-04-13-daynightchart-responsive-design]] 的关键思想是：

> 5min/格始终不变，响应式只改变 rowsPerCol。

这对任务排程非常重要。因为任务估时最终都要转换成 5 分钟格：

```text
estimatedMinutes = 25 -> 5 格
estimatedMinutes = 45 -> 9 格
estimatedMinutes = 90 -> 18 格
```

因此任务调度层应该依赖同一套时间颗粒，而不是自己发明 15/30/60 分钟块。

如果后续实现 superpowers 规格，任务 overlay 也要跟着 `rowsPerCol` 自适应。

## 第一轮实现建议

如果继续进入代码实现，我建议按这个顺序：

1. 先强化 `DayNightChart` 内部的“昼夜作战台”，把编辑入口升级为规划/记录控制台。
2. 让拖拽填格体验服务两个语义：过去记录和未来规划。
3. 再定义任务进入 5 分钟格的前端结构。
4. 等交互稳定后，再把 mock / 前端状态替换为 Rust SQLite 持久化。
5. 最后再考虑动机、奖励、Buff 是否拆成主界面的其它面板。

为什么先前端 mock：

- 当前核心风险不是数据库，而是主页面到底长什么样。
- 动机仪表盘必须先定魂。
- 任务进入昼夜表的交互必须先看见，才能判断字段是否足够。

## 不要做的事

- 不要把 `src/lib/game` 旧模型直接搬上首页。
- 不要先做复杂等级、属性、技能树。
- 不要先做开箱、合成、稀有度。
- 不要把人升的主结构照搬过来。
- 不要让“暗影君主系统面板”重新成为首页。
- 不要让 Fairy 只做闲聊，它必须能读任务、时间、奖励和动机。

---

## 本轮结论

下一阶段最小闭环应该是：

```text
动机 -> 目标 -> 任务 -> 昼夜表排程 -> 感知/手动验收 -> 奖励流水 -> 商店/仓库 -> 成就/Buff -> Fairy 下一轮更懂你
```

这条线跑通以后，暗影君王系统才会从“好看的时间追踪器”变成真正的 AI 第二系统。
