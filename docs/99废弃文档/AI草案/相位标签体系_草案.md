# 相位标签体系（草案）

> 状态：草案。  
> 这份文档用于定义“日内相位引擎”在界面上的命名方式。核心判断：底层可以叫“相位”，但前台不应该一直展示抽象的“相位”二字，而应该用更有系统感、更像运行规则的“协议”。

参考文档：[[日内相位引擎_草案]]、[[第3章_系统面板设计]]、[[第4章_今日协议AI驱动]]、[[主界面编排方案_多面板系统桌面]]、[[产品考古索引]]、[[管制与惩罚]]、[[主动询问与窗口记忆]]。

---

## 1. 总体命名原则

产品内部可以这样分层：

```text
底层概念：日内相位
前台显示：系统协议
具体协议：韧性 / 执行 / 发现 / 回收 / 保护
```

这样做是为了避免用户界面显得过于抽象。

“相位”适合写在文档、代码、数据模型里，因为它描述的是系统状态；但用户每天看到的东西应该更像系统正在运行的一套规则：

```text
当前协议：韧性
当前协议：执行
当前协议：发现
当前协议：回收
当前协议：保护
```

协议比相位更有行动含义。它暗示：

```text
当前允许什么
当前禁止什么
当前优先级是什么
Fairy 该用什么语气
设备管制是否开启
哪些面板应该显示
```

---

## 2. 主标签页命名

当前主舞台不应该叫：

```text
动机仪表盘
```

更合适的标签是：

```text
今日协议
```

理由：

- 动机不应该全天常驻前台。
- 前台真正要回答的是“今天此刻系统以什么协议运行”。
- 这个标签能自然承载每日任务、设备管制、语境入口、复盘、保护等不同内容。
- 它比“动机仪表盘”更适合成为系统首页。

主舞台第一版可以是：

```text
今日协议 / 昼夜表
```

后续扩展可能是：

```text
今日协议 / 昼夜表 / 奖励 / 记忆
```

但不建议一开始就做太多主标签。

## 3. 今日协议与当日协议

顶部栏已有日期选择。这个日期应该同时影响：

```text
昼夜表
协议页
右侧详情 / Fairy 上下文
```

因此，“今日协议”这个标签不是固定名称，而是随所选日期变化。

规则：

```text
selectedDate 是今天 -> 今日协议
selectedDate 不是今天 -> 当日协议
```

示例：

```text
顶部日期：2026-05-20
如果今天就是 2026-05-20：
  标签显示：今日协议
  含义：正在运行中的系统协议

顶部日期：2026-05-16
如果今天是 2026-05-20：
  标签显示：当日协议
  含义：回看 2026-05-16 当天系统如何运行
```

这个命名比“历史协议”更好，因为用户不是在看一条历史记录，而是在看那个日期自身的一套日内运行结构。

一句话：

> 今天是活的，所以叫今日协议；历史日期是被封存的一天，所以叫当日协议。

## 4. 日期联动模型

顶部日期不应该只控制昼夜表。它应该成为“当前查看日期”的全局锚点。

```text
selectedDate
  ↓
DayNightChart 显示该日实际记录 / 计划安排
  ↓
ProtocolPanel 显示该日协议状态
  ↓
FairyContext 注入该日任务、语境、奖励、管制状态
```

今天与历史日期的区别：

```text
今天：
  协议是 live 的
  可以改变当前相位
  可以触发设备管制
  可以安排未来时间格
  Fairy 可以主动干预

历史日期：
  协议是 archived 的
  主要用于回看和复盘
  不应该触发设备管制
  不应该改变当下运行状态
  Fairy 主要解释那天发生了什么
```

因此，协议页需要支持两种状态：

```text
LiveProtocolView
ArchivedProtocolView
```

它们可以共用同一个界面骨架，但操作权限不同。

## 5. 五个核心协议标签

### 5.1 韧性协议

内部 key：

```text
resilience
```

前台名称：

```text
韧性协议
```

短标签：

```text
韧性
```

英文 HUD：

```text
RESILIENCE
```

使用场景：

```text
早晨
睡醒后重启
状态很散时
基础任务未完成时
需要先立秩序时
```

前台内容：

```text
每日基础任务
身体启动
晨间整理
设备管制
硬截止摘要
```

Fairy 话术方向：

```text
先把身体和秩序立起来。
探索下午再开放。
今天先赢下最小的一场。
```

### 5.2 执行协议

内部 key：

```text
execution
```

```text
执行协议
```

```text
执行
```

```text
EXECUTION
```

```text
主线工作
```

### 5.3 发现协议

### 5.4 回收协议

内部 key：

```text
recovery
```

前台名称：

```text
回收协议
```

短标签：

```text
回收
```

英文 HUD：

```text
RECOVERY
```

使用场景：

```text
晚上
工作结束后
探索结束后
需要复盘和整理时
```

前台内容：

```text
昼夜表复盘
实际记录 vs 计划安排
语境小球整理
动机提炼
Buff 沉淀
奖励结算
明日预排
```

Fairy 话术方向：

```text
今天哪个语境反复发光？
今天更靠近了什么？
今天逃避了什么？
明天早晨不要继续想这个，先做韧性任务。
```

注意：

“回收”不是“恢复”。这里的意思是把一天散落的行动、语境和情绪回收成可用资产。

## 6. 顶部栏文案

顶部栏可以逐渐从“日期选择”变成“日期 + 协议态”的组合。

基础版本：

```text
日期  05/20 周三  今天
系统状态  ONLINE
```

协议版本：

```text
日期  05/20 周三  今天
当前协议  韧性
系统状态  ONLINE
```

如果所选日期不是今天：

```text
日期  05/16 周六
当日协议  已归档
系统状态  ARCHIVE VIEW
```

不建议把顶部栏做得过重。顶部只显示当前日期和协议摘要，详细内容交给“今日协议 / 当日协议”标签页。

## 7. 协议页内部标签

`今日协议` 页内部可以分几块，但第一版不一定都实现。

建议标签：

```text
当前协议
任务许可
设备管制
语境入口
Fairy 指令
```

今天显示：

```text
当前协议：韧性协议
剩余窗口：42min
解锁条件：完成晨间基础任务
设备管制：娱乐锁定
发现入口：未开放
Fairy 指令：先把身体和秩序立起来。
```

历史日期显示：

```text
当日主协议：执行 / 回收
协议切换：韧性 07:20 -> 执行 10:05 -> 发现 17:30 -> 回收 21:10
设备管制记录：3 次
发现入口开启：1 次
Fairy 当日判断：下午发现时间过长，压缩了晚间回收。
```

## 8. 时间戳存档

后续需要把协议变化按时间戳存档。

最小概念：

```ts
interface ProtocolSpan {
  id: string
  date: string
  phase: 'resilience' | 'execution' | 'discovery' | 'recovery' | 'protection'
  startAt: string
  endAt: string | null
  source: 'time_default' | 'manual' | 'fairy' | 'rule' | 'state_override'
  reason: string
  devicePolicy: 'none' | 'soft_lock' | 'hard_lock'
  discoveryAllowed: boolean
  createdAt: string
}
```

可以先不建表，但设计上要承认：

```text
协议不是当天的一个值，而是一条时间线。
```

这条时间线未来可以叠加到昼夜表上：

```text
07:00 - 10:00 韧性
10:00 - 17:30 执行
17:30 - 20:30 发现
20:30 - 23:30 回收
23:30 - 05:00 保护
```

它会解释为什么某些内容在某个时间被允许，为什么某些设备在某个时间被管制。

## 9. 旧命名替换建议

### 主舞台标签

当前：

```text
动机仪表盘
昼夜表
```

建议：

```text
今日协议 / 当日协议
昼夜表
```

### MotivationDashboard

组件名可以暂时不急着改，但语义上它应该被重新定位。

当前：

```text
MotivationDashboard = 动机仪表盘
```

建议未来变成：

```text
ProtocolDashboard = 今日协议面板
```

其中动机卡片只在回收、复盘、偏离解释时出现，不作为全天默认内容。

### ViewSwitcher

文案动态化：

```ts
motivation label:
  isSelectedDateToday ? '今日协议' : '当日协议'
```

注意：内部 enum 可以先不急着改，避免一次重构过大。但 UI 文案应该先改。

## 10. Fairy 话术标签

Fairy 在对话里应该少说抽象名词，多说当前协议。

不推荐：

```text
你现在处于 resilience phase。
```

推荐：

```text
现在是韧性协议，不开放发现流。
先完成晨间基础任务，下午再让你探索。
```

不推荐：

```text
你的动机进度下降。
```

推荐：

```text
今天不是审判动机的时候。先把这 25 分钟执行完，晚上回收协议里我再帮你整理。
```

不推荐：

```text
你应该复盘一下。
```

推荐：

```text
进入回收协议。把今天发光的三个语境收起来，明天不用从头想。
```

## 11. 当前结论

第一批推荐标签如下：

```text
主标签：
今日协议 / 当日协议
昼夜表

协议：
韧性协议
执行协议
发现协议
回收协议
保护协议

协议短标签：
韧性
执行
发现
回收
保护

关键状态：
设备管制
发现入口
语境小球
Fairy 指令
当日归档
协议切换
```

这一套命名的核心作用：

```text
动机退到后台。
协议站到前台。
时间成为界面的主索引。
日期同时驱动昼夜表和协议页。
今天是运行态，历史日期是归档态。
```

如果这套成立，下一步可以先做很小的 UI 文案调整：

```text
动机仪表盘 -> 今日协议 / 当日协议
```

不急着改数据模型，不急着重写组件。先让产品语言回到正确轨道。
