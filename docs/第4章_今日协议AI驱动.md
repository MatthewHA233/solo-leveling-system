# 第4章 今日协议：AI 驱动的日内运行系统

> 本章承接 [[第1章_产品愿景]]、[[第3章_系统面板设计]]、[[日内相位引擎_草案]] 和 [[相位标签体系_草案]]。  
> 核心判断：`今日协议 / 当日协议` 不是手动维护的仪表盘，而是由 Fairy 和系统上下文共同生成、解释、归档的 AI 运行层。

![[今日协议AI驱动概念图.png]]

> 概念图：`今日协议` 位于中心，围绕它运转的是五类协议、昼夜表、语境小球、设备管制、Fairy 和日期归档。它不是静态仪表盘，而是系统根据时间、任务、感知层和用户规则生成的当日运行层。

---

## 4.1 为什么必须 AI 驱动

`今日协议` 如果只是一个用户手动填写的页面，就会退化成另一种待办面板。

真正的协议页应该回答的是：

```text
现在系统为什么这样运行？
此刻应该开放什么，限制什么？
用户下一步最该做什么？
哪些语境应该出现，哪些应该被推迟？
如果用户想突破限制，系统如何询问和记录？
这一天结束后，系统如何解释当天发生了什么？
```

这些问题不能靠固定 UI 配置解决，因为它们依赖实时上下文：

```text
现在几点
选中的日期是不是今天
今天是否还有硬 deadline
基础任务是否完成
昼夜表上剩余多少 5 分钟格
过去几小时实际做了什么
用户是否离席、疲劳、偏离或高刺激输入过多
是否有新的语境片段值得回收
用户此前给过哪些规则和纠偏
```

因此，`今日协议` 的本质不是“页面”，而是 AI 对当天运行状态的生成结果。

一句话：

> 用户不维护协议，AI 生成协议；用户只确认、纠偏、授权和突破。

---

## 4.2 今日协议与当日协议

顶部日期是整个系统的时间锚点。它同时影响 [[第3章_系统面板设计|昼夜表]] 和协议页。

```text
selectedDate 是今天 -> 今日协议
selectedDate 不是今天 -> 当日协议
```

### 今日协议

`今日协议` 是 live 状态。

它描述当前日期正在运行的系统协议：

```text
当前协议是什么
为什么进入这个协议
当前允许什么
当前限制什么
下一步行动是什么
解锁条件是什么
Fairy 当前判断是什么
```

它可以影响现实：

```text
开启或关闭设备管制
限制发现流
推荐任务进入昼夜表
要求用户说明突破原因
调整 Fairy 的说话方式
```

### 当日协议

`当日协议` 是 archive 状态。

它不是让历史日期继续管制用户，而是解释那一天曾经如何运行：

```text
当天有哪些协议切换
哪些时段执行成功
哪些时段偏离
哪些管制被触发
哪些语境被收集
哪些任务被排入昼夜表但没有完成
Fairy 对那一天的复盘判断
```

它不应该触发当下设备管制，也不应该改变今天的运行状态。

一句话：

> 今日协议负责运行，当日协议负责解释。

---

## 4.3 AI 的三层职责

今日协议至少需要三层 AI。

### 协议判定 AI

负责判断当前应该进入哪个协议。

输入：

```text
当前时间
顶部所选日期
当天任务
实际记录
计划安排
最近一段时间感知层
用户规则
设备状态
疲劳/离席/高刺激输入信号
```

输出：

```text
当前协议
置信度
判定原因
是否需要询问用户
是否需要触发管制
```

示例：

```text
当前协议：韧性协议
原因：当前是早晨，基础任务未完成，昨晚回收不足，发现入口不应提前开放。
置信度：0.82
需要询问：否
设备管制：开启娱乐软锁
```

### 协议解释 AI

负责把判定结果转成用户能理解、能执行的前台内容。

输出：

```text
一句系统判断
当前允许
当前限制
下一步
解锁条件
Fairy 话术
```

示例：

```text
系统判断：
先把身体和秩序立起来，下午再开放发现。

当前允许：
洗漱、吃饭、晨间整理、基础训练、低刺激音乐。

当前限制：
短视频、游戏、发现流、复杂动机分析。

下一步：
12 分钟内完成洗漱，然后进入 15 分钟晨间整理。

解锁条件：
完成 2 个基础任务后，下午发现协议可开放。
```

### 协议归档 AI

负责在一天结束或用户回看历史日期时，把当天协议时间线解释成档案。

输出：

```text
当日主线
协议切换时间线
关键偏离
关键完成
语境收集
管制记录
Fairy 复盘
明日策略建议
```

示例：

```text
当日主线：
上午韧性协议完成较晚，导致执行协议启动延后；下午发现协议提前打开，压缩了晚间回收。

协议切换：
08:20 韧性 -> 11:05 执行 -> 16:40 发现 -> 21:30 回收 -> 01:10 保护

Fairy 复盘：
这天不是意志力不足，而是早晨韧性协议没有及时完成，后续全线松动。
```

## 4.4 协议页的输入上下文

今日协议需要一个明确的上下文快照，而不是让 AI 临时猜。

最小上下文：

```ts
interface ProtocolContextSnapshot {
  now: string
  selectedDate: string
  isToday: boolean
  currentTimeMinute: number

  currentProtocolSpan: ProtocolSpan | null
  protocolHistory: ProtocolSpan[]

  dayNight: {
    actualBlocks: unknown[]
    plannedBlocks: unknown[]
    remainingMinutes: number
    freeWindows: Array<{ startMinute: number; endMinute: number }>
  }

  tasks: {
    active: unknown[]
    plannedToday: unknown[]
    overdue: unknown[]
    hardDeadlines: unknown[]
    foundationTasks: unknown[]
  }

  perception: {
    recentApps: unknown[]
    recentTags: unknown[]
    recentBili: unknown[]
    presenceState: 'present' | 'absent' | 'unknown'
    highStimulationSignals: unknown[]
  }

  contextMemory: {
    recentBubbles: unknown[]
    activeBuffs: unknown[]
    pendingRecoveryItems: unknown[]
  }

  userRules: {
    devicePolicies: unknown[]
    phasePreferences: unknown[]
    correctionHistory: unknown[]
  }
}
```

这里的类型可以后续再细化。第一版重要的不是字段完美，而是确认协议页必须由一份上下文快照驱动。

协议页不能只读“任务列表”。它至少要同时读：

```text
时间
日期
昼夜表
任务
感知层
语境
设备规则
用户纠偏历史
```

## 4.5 协议页的输出结构

AI 最终应该生成一个可渲染、可归档、可纠偏的对象。

```ts
interface ProtocolBrief {
  date: string
  mode: 'live' | 'archive'

  phase: 'resilience' | 'execution' | 'discovery' | 'recovery' | 'protection'
  phaseLabel: '韧性协议' | '执行协议' | '发现协议' | '回收协议' | '保护协议'

  headline: string
  reason: string
  confidence: number

  allowed: string[]
  restricted: string[]
  nextAction: {
    title: string
    estimatedMinutes: number | null
    deadlineMinute: number | null
  } | null

  unlockConditions: string[]
  devicePolicy: 'none' | 'soft_lock' | 'hard_lock'
  discoveryAllowed: boolean

  fairyLine: string
  userActions: ProtocolUserAction[]
}

type ProtocolUserAction =
  | 'accept'
  | 'delay'
  | 'switch_protocol'
  | 'explain'
  | 'request_breakthrough'
  | 'correct_fairy'
  | 'remember_rule'
```

前端页面不应该直接让用户编辑 `reason`、`allowed`、`restricted` 这些内容。用户应该通过动作影响下一次生成。

例如：

```text
用户点击：纠正 Fairy
用户输入：今天早晨不是拖延，是有突发电话
系统记录 correction
下一次协议生成时把这条纠偏纳入上下文
```

## 4.6 用户交互不是编辑，而是纠偏

今日协议的用户动作应该很少，但很关键。

### 接受

表示用户接受当前协议和下一步。

系统记录：

```text
用户接受了当前协议
协议继续运行
如果有下一步任务，可进入昼夜表计划块
```

### 推迟

表示用户暂时不执行，但必须给出时间或原因。

Fairy 不应该接受含糊的“待会儿”。

```text
用户：待会儿
Fairy：具体几点？如果不给时间，我不能把协议让出去。
```

### 切换协议

用户可以主动切换协议，但系统要记录原因。

```text
从 韧性协议 切到 发现协议
原因：用户手动切换
说明：今天早上已有外出运动，允许提前探索
```

### 申请突破

用于突破设备管制、发现入口或保护协议。

示例：

```text
你要突破保护协议去看视频，原因是什么？

A. 和任务有关
B. 情绪恢复
C. 只是想逃避
D. 其他
```

突破不一定禁止，但必须被记录。记录本身就是系统的管制感。

### 纠正 Fairy

Fairy 可能判断错。用户必须能纠偏，否则 AI 会变成强势但不可信的管制者。

纠偏应该进入长期记忆：

```text
某些窗口不是娱乐，而是研究材料
某些时间段用户确实需要休息
某类任务早晨不适合做
某个设备限制太强，需要降级
```

### 记住规则

当用户多次纠偏同一类判断时，Fairy 可以询问是否写入规则：

```text
你已经第三次说明“下午 16:00 后允许探索类输入”。
要把它写入发现协议规则吗？
```

## 4.7 协议时间戳与归档

协议不是当天的一个静态值，而是一条时间线。

```ts
interface ProtocolSpan {
  id: string
  date: string
  phase: 'resilience' | 'execution' | 'discovery' | 'recovery' | 'protection'
  startAt: string
  endAt: string | null
  source: 'time_default' | 'manual' | 'fairy' | 'rule' | 'state_override'
  reason: string
  confidence: number
  devicePolicy: 'none' | 'soft_lock' | 'hard_lock'
  discoveryAllowed: boolean
  createdAt: string
}
```

示例：

```text
2026-05-20
07:15 - 10:05 韧性协议
10:05 - 16:40 执行协议
16:40 - 19:20 发现协议
19:20 - 23:35 回收协议
23:35 - 02:10 保护协议
```

这条时间线未来可以叠加到昼夜表：

```text
昼夜表显示实际做了什么
协议时间线显示系统当时认为应该怎样运行
二者对比产生复盘价值
```

当用户查看历史日期时，AI 不只是显示旧 span，而是生成当日解释：

```text
协议计划和实际行为是否匹配
哪个协议被挤压
发现是否提前打开
保护是否失败
韧性失败是否影响全天
```

这就是 `当日协议`。

## 4.8 与昼夜表的关系

昼夜表回答：

```text
这一天每 5 分钟发生了什么，或计划发生什么。
```

今日协议回答：

```text
系统为什么允许这些事发生，为什么限制另一些事发生。
```

二者必须共享同一个日期。

```text
顶部日期改变
  ↓
昼夜表切换到那一天
  ↓
协议页切换到那一天
  ↓
Fairy 解释也切换到那一天
```

今天：

```text
昼夜表可以继续填未来计划
今日协议可以继续影响接下来的设备管制和内容入口
```

历史：

```text
昼夜表用于回看实际记录
当日协议用于解释那天系统如何运行
```

协议也可以反向影响昼夜表：

```text
执行协议 -> 鼓励把任务排进未来格
回收协议 -> 鼓励对实际记录做解释
保护协议 -> 不鼓励继续规划复杂任务
发现协议 -> 允许打开语境小球，而不是强推执行
```

## 4.9 与语境小球的关系

语境小球不是全天开放的内容流。

它主要出现在两个协议里：

```text
发现协议：收集
回收协议：整理
```

发现协议里，语境小球是入口：

```text
一段视频
一张截图
一次聊天
一个窗口片段
一个突然发光的想法
```

回收协议里，语境小球是材料：

```text
这段语境为什么发光？
它关联哪个动机？
它能不能形成 Buff？
它有没有生成一个任务？
它是不是只是刺激？
```

今日协议决定：

```text
现在是否允许打开语境流
是否允许刷短视频式回看
是否应该把语境暂存到晚上
是否应该立即打断并进入执行
```

这可以避免“语境系统”变成新的分心入口。

## 4.10 与设备管制的关系

设备管制不是独立惩罚模块，而是协议的执行手段之一。

不同协议下管制强度不同：

```text
韧性协议：
  娱乐设备默认软锁或硬锁
  发现流关闭

执行协议：
  高刺激应用限制
  与任务相关应用允许

发现协议：
  内容输入开放，但有时间边界
  系统记录收集目的

回收协议：
  关闭无限探索
  只允许整理已收集语境

保护协议：
  高刺激应用强限制
  保留睡眠、必要通信和明日安心卡
```

设备管制必须可解释、可申请突破、可被纠偏。

如果系统只锁设备但不解释，就会变成粗暴工具；如果系统只解释但不执行，就没有管制感。

今日协议正好把两者接起来：

```text
AI 解释为什么
系统执行限制
用户可以申请突破
突破进入归档
归档影响未来规则
```

## 4.11 与 Fairy 的关系

Fairy 是今日协议的人格化解释器。

但 Fairy 不应该凭空说话。它每次回应前都应该知道当前协议：

```text
当前是哪个协议
为什么进入该协议
当前允许和限制
下一步动作
用户是否正在突破
今天是否 live
当前查看日期是否历史
```

同一句话，在不同协议里应该不同。

用户说：

```text
我想看会儿视频。
```

韧性协议下：

```text
不开放。先完成基础任务，下午发现协议再说。
```

执行协议下：

```text
如果是任务材料，可以说明用途；如果是逃避，我会锁掉。
```

发现协议下：

```text
可以，但给它一个收集目的。看完后收成一个语境小球。
```

回收协议下：

```text
不要再开新输入。把今天已经看过的东西回收掉。
```

保护协议下：

```text
现在不看。保存现场，明天处理。
```

这就是协议对 Fairy 的意义：它不是给 UI 换标签，而是改变 AI 的判断方式。

## 4.12 第一版实现边界

第一版不需要做完整自主智能。

可以先做：

```text
1. UI 文案：动机仪表盘 -> 今日协议 / 当日协议
2. 时间默认：按当前时间给一个默认协议
3. Mock brief：前端先展示 AI brief 的样子
4. 用户动作：接受 / 推迟 / 解释 / 纠正
5. 归档概念：先在文档和类型里保留 ProtocolSpan
```

暂缓：

```text
真实设备锁定
复杂多模型 Agent
自动生成语境小球
完整 reward 结算
复杂长期记忆规则
```

关键是先把产品语言和交互重心扳正：

```text
用户打开系统，不是看到一堆动机卡片。
用户看到的是：此刻系统以什么协议运行，为什么，下一步是什么。
```

## 4.13 本章结论

今日协议是暗影君王系统从“好看的时间追踪器”走向“AI 第二系统”的关键层。

它把这些东西连接起来：

```text
时间
昼夜表
任务
语境
动机
设备管制
Fairy
归档复盘
```

它的核心不是展示，而是判断。

```text
AI 判断当前协议
AI 解释当前协议
AI 记录协议变化
AI 在历史日期生成当日协议档案
用户纠偏并授权
系统把纠偏变成未来规则
```

最终目标：

> 在正确的时间，把正确的自己叫出来；在错误的时间，把错误的入口关掉。

这就是 `今日协议 / 当日协议` 应该承担的角色。
