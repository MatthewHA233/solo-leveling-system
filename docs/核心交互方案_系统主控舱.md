## 为什么不能是按钮

右上角按钮在语义上通常代表：

```text
设置
工具
次级入口
可有可无的附属功能
```

但动机仪表盘、昼夜表任务调度、奖励经济和 Fairy 解释层不是附属功能。它们是系统本体。

所以核心交互不应该是：

```text
用户想起来了 -> 点一个小按钮 -> 看到系统
```

而应该是：

```text
系统一直在场 -> 根据现实状态发出指令感 -> 用户展开主控舱 -> Fairy 解释、重排、裁决
```

## 科幻作品里的交互隐喻

更接近的隐喻不是“App 的导航按钮”，而是：

- 飞船舰桥的主控台。
- 钢铁侠 HUD 中央反应堆。
- EVA 作战指挥屏。
- 赛博朋克任务简报层。
- 游戏里的系统指令面板。

它们有几个共同点：

- 常驻：核心状态一直可见。
- 分层：平时是摘要，展开是完整指令。
- 带判断：不是罗列信息，而是告诉你“现在该怎么处理现实”。
- 视觉上有重量：它占据主界面的仪式性位置，不和设置按钮同级。
- 能接管注意力：当情况紧急或任务冲突时，它可以主动升级显示。

## 新交互结构

### 1. SYSTEM CORE DECK

旧方案设想：主控舱常驻在顶栏下方，位于昼夜表上方。

它不是弹窗，也不是侧栏按钮，而是系统的“呼吸条”：

```text
┌──────────────────────────────────────────┐
│ 顶栏：日期 / 状态 / 模型 / 设置           │
├──────────────────────────────────────────┤
│ SYSTEM CORE DECK                         │
│ 动机核 | 今日指令 | 奖励回路 | 展开状态   │
├───────────────────────┬──────────────────┤
│ 昼夜表                 │ 系统面板 / Fairy  │
└───────────────────────┴──────────────────┘
```

常驻摘要包含：

- `SYSTEM CORE`：当前最重要动机，以及 Fairy 的一句判断。
- `TODAY DIRECTIVE`：当前最该处理的任务，以及还剩多少 5 分钟格没有落地。
- `REWARD LOOP`：钱包、最近奖励、激活 Buff。
- `CORE ONLINE / OPEN CORE`：展开状态，不是普通按钮，而是主控舱状态。

### 2. 右侧主控面板

当 Core Deck 展开时，右侧显示 `SystemPilotPanel`。

它不是“另一个页面”，而是主控舱的展开态：

```text
动机仪表盘
今日任务装载情况
奖励流水和钱包
Buff 记忆
Fairy 的解释
```

当 Core Deck 收束时，右侧回到 Fairy Chat。

这意味着：

```text
Core Deck = 系统一直在场
SystemPilotPanel = 系统展开解释
ChatPanel = Fairy 通讯频道
```

### 3. 细节优先级

当用户 hover 或 pin 昼夜表中的具体 span 时，右侧细节面板优先显示。

优先级：

```text
昼夜表细节 > 编辑模式标签库 > Core 展开面板 > Fairy Chat
```

这是合理的，因为用户已经在“现实证据”上指认了一个具体时间段，系统应该先解释那个证据。

## 交互状态机

```text
Ambient Core
  系统核心常驻摘要，显示动机/指令/奖励。

Expanded Core
  右侧展开完整系统面板，显示动机仪表盘和任务经济闭环。

Evidence Focus
  用户悬停或固定昼夜表 span，右侧临时进入证据解释。

Fairy Channel
  Core 收束后右侧回到聊天，Fairy 负责对话和重排。

Command Escalation
  当任务装不进今日、偏离主线、奖励异常、证据不足时，Core Deck 可以高亮或提示。
```

---

## 为什么这比按钮好

按钮方案的问题：

- 系统核心被降级。
- 用户需要主动想起它。
- 动机和任务没有持续存在感。
- 看起来像一个“功能页面”，不是第二系统。

主控舱方案的好处：

- 动机一直在现实时间上方。
- 今日任务永远和剩余时间绑定。
- 奖励回路不是隐藏结算，而是系统心跳。
- Fairy Chat 不再是全部，而是系统的一条通讯频道。
- 视觉上更接近“我被一个系统守望着”，而不是“我打开了一个小工具”。

## 设计原则

- 核心状态常驻，不藏入口。
- 展开不是跳页，而是主控舱升起。
- 所有任务必须能回到动机。
- 所有任务都必须能换算成 5 分钟格。
- 所有奖励必须进入流水账本。
- Fairy 的话必须引用当前动机、剩余时间、证据或 Buff。
- 系统视觉可以科幻，但交互目的必须现实。

最终目标：

> 让用户感觉不是“打开了任务管理器”，而是“进入了自己的作战舰桥”。
